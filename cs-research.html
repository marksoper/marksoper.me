Data-Focused UI/UX Design

Tuesday
July 13
, 2015

Can user-acceptable web UI implementations be derived from an application's complete production data model (as defined to include both the transactional database schema and a the event definitions associated with a properly-instrumented analytics database)?


Two Problems with Typical UI/UX Design As It Is Currently Done

New application design typically means producing a visual representation of UI/UX  (wireframes, mocks, clickable prototype) that serves as requirements for the subsequent development phase.

There are two major problems with this typical approach:

  1. As development evolves, the product increasingly deviates from the design, meaning there’s no longer a definitive specification that product owners, developers, designers, QA, etc. can use for a shared understanding of the product. One symptom of this problem is that there isn’t a single consistent and precise notion of the product that can be used in instrumenting for analytics (reasoning about the production system for business and technical purposes). The team may lack a common language for referring to the various screens, components, features, etc. involved in the UI, making it hard to map analytics events back to specific behaviors the user executed in the product.
  2. The typical visual forms (wireframes, mocks, clickable prototype) require the team to sort out details of the UI that aren't critical to early user acceptance. Much effort goes into decisions about layout, navigation, aesthetics, and lots of other aspects that may not improve user experience. As an example, consider a team deciding between (a) the user composes a new message within the visual context of the message thread, or (b) the user is taken to a different screen for composing new messages. In my experience this kind of feature can be a source of lot of lateral movement in the project - debates among the team, prototyping it one way and then the other way - that cost valuable time. Often the right answer isn’t knowable without user testing and the right answer varies with screen-size: (a) is right for desktop while (b) is right for smartphone. Ideally the initial design would deal with this issue at a higher level: “users must be able to compose new messages” and leave the rest to implementation.

A Proposed Data-Focused Approach To UI/UX Design

Let’s consider a different approach that solves these problems and has the potential for big gains in speed to implement, quality, and malleability of the resulting application.

Step 1: Visual UI/UX Design

Similar to the first step in the traditional approach, design begins with product owner, developer, and designer working together to get the product vision out of the owners’ heads and into a visual representation of the UI and UX. The difference is that this step needn’t have the rigor required when the goal is to produce a requirements specification. The team can deal with a high-level representation of the product and work quickly by deferring details to later. Digital tools like Invision can used, but whiteboard, pen and paper, simple wireframes should be sufficient.

Step 2: Software System Design

Using the output of Step 1 as a guide, the team works together to produce a functional software design specification for the system in the format of a data model and a set of user/system behaviors. The goal here is to specify system data and behavior at the most fundamental level. Consider an example:

The system involves an entity called User which has the following properties and relationships to other entities ...

The system involves an entity called Message which has the following properties and relationships to other entities …

User must be able to create a new Message

User must be able to send a Message to another User

User must be able to view a received Message

This approach is premised on the assumption that for a certain class of applications (new development, CRUD heavy, basic UI requirements), the product owner may be reasonably satisfied with ANY application UI and UX that conforms to all relevant constraints as stated in the form above. A rapidly developed application that simply meets the basic functional requirements may not be sufficient in the long-term, but the near-term benefits can be huge: it allows the owner to experiment with the product and start end-to-end user testing right away to challenge and improve upon initial design assumptions.

Notice that this design format has two sections that should seem familiar:

Defining entities - this is classic data modeling a la Rails Active Record and other such systems

Defining system and user behaviors - this is roughly equivalent to the process that the team goes through in instrumenting the product for analytics. Instrumentation is where the team decides which “events” relevant to the system/prodcut it deems important enough to track and analyze for business and technical purposes. Such events might be either user-initiated (User sends a Message) or system-initiated (System sends an account sign-up confirmation email to a User).

So if this theory (that the product owner can be initially satisfied with a rapidly-generated app that conforms to the fundamental design rules) holds true, we have an interesting result:

Transaction data model and analytics-instrumentation events provide sufficient design specification for an entire application software system, at least at a prototype stage. In other words, describing the data that powers a system is sufficient to describe that system at a fundamental, functional level.

This is a very powerful result because:

  1.  It means that the team can create and evolve the application by managing a set of domain-specific rules written in a high-level language that business people can use.
  2. With the right middleware (which doesn’t yet exist), UI software can be reliably and immediately generated from these rules. This means that application UI can always be kept precisely synchronized with requirements AND with analytics instrumentation (since this IS the design spec)
